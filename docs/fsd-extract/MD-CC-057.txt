Process Specification
In this section you can find detailed business process description and the corresponding workflows.
Business Process
  What?
The Home Dashboard consolidates operational, financial, and agronomic data from all farmers assigned to the logged user's specific hierarchical node. The visibility of the dashboard cards is dynamic and strictly coupled with the lifecycle defined in the crop calendar (Crop Calendar).
  Who?
Agri360 Users.
  When?
Used on a daily basis as home page for login.
  Where?
Agri360 (Home/Dashboard Screen).
  Why?
To centralize key KPIs into a single, unified view and deliver an intuitive, frictionless entry point for all operational sub-modules without cluttering the main stream area.
  How?
The system renders dashboard cards based on the Crop Calendar status.
User Interface & User Experience
In this section you can find details on Screens, their purpose, expected navigation behavior, CRUD processing description and Data Requirements. Furthermore, you can also find Business Rules, their triggers, corresponding validations, and details on error handling.
4.1 General Architecture & Shared Business Rules
Dynamic Card Rendering (Crop Calendar Dependency):
If a crop season's contracting phase is Active in the calendar -> Render the Contracting Card.
If any crop season is Active -> Render the Field Notebook Card.
If a crop season's commercialization phase is Active -> Render the Commercialization Card (Subject to the Hold rule).
Tobacco Type Tabs (CultureType): All metric cards (Contracting, Commercialization, and Field Notebook) must provide horizontal tab navigation to segregate data by tobacco type (e.g., FCV, BLY, AM), dynamically filtering the consolidated values based on the selection.
User Hierarchy Restriction: The dashboard and search entity results must strictly isolate and consolidate data from clusters and farmers belonging exclusively to the hierarchical tree of the logged-in user.
4.2 List of Screens - Web
4.2.1 SCREEN 01: Dashboard - Home (Main View)
Screen Purpose: The initial screen of the mobile app that presents the general data consolidation of the User’s hierarchical node.
4.2.2 SCREEN 02: Card - Contracting (Contracting Phase)
Screen Purpose: To monitor the progress of contracting targets and the compliance/credit profile of the user’s hierarchical node for the card’s crop season.
Data Fields & KPI Specifications:
Production Target (Kg):
Metric: Displays a progress bar comparing the total volume contracted against the target.
Formula:
Contracted Farmers:
Metric: Displays the number of farmers with a signed contract for this crop season vs. the total active farmers in the hierarchical branch.
Active Farmer Rule: A farmer who held an active contract in the immediately preceding crop season.
Segmentation: A graphical representation (Donut Chart) breaking down contracted farmers by their classification/segmentation (e.g., S2, S3, S4 or future Diamond, Gold, Silver, Bronze).
Variety Profile:
Metric: A vertical bar chart comparing the planted area target (Target in Hectares) per seed variety (e.g., BAT 1009, BAT 2009) vs. the total quantity of seeds commercialized/distributed.
Approved Pre-Assessment (ESG):
Metric: A simple counter displaying the absolute number of farmers who have completed the ESG pre-assessment and are flagged with an Approved status for the current crop cycle.
Indebtedness Level:
Metric: The projected average percentage of debt for all contracted farmers in the current season.
Calculation Formula:
4.2.3 SCREEN 03: Card - Field Notebook (Field Notebook)
Screen Purpose: To centralize the completion progress of the field notebooks across all farmers for this hierarchical tree branch.
Data Fields & KPI Specifications:
General Metric per Topic: Displays the completion percentage for each agricultural stage (Seedbed: Sowing/Fertilizer/Pesticide/Pest Occurrence, Field: Planting/Organic Fertilizer/Pest Occurrence, Topping, Harvest).
Metric Business Rule: The calculation is tracked at the Cluster level. If a Cluster has at least one (1) record submitted in the corresponding form (e.g., Sowing), that cluster is marked as "Completed" for that specific topic.
Formula:
4.2.5 SCREEN 05: Card - Commercialization (Commercialization Phase) - [STATUS: ON HOLD]
Screen Purpose: To provide financial visibility and track the intake flow of tobacco during the commercialization season.
CRITICAL BUSINESS RULE: This card and all its internal sub-components are strictly classified as "ON HOLD". The development of this user interface and its underlying data integrations is deferred until the technical Discovery for the BUYING user stories is completed and delivered.
Data Fields & Specification (Draft for Future Implementation):
Delivered Tobacco (Kg):
Completed Contracts:
% Tobacco Flow X % Debt Recovery: A timeline chart crossing the physical intake flow of tobacco against real financial debt amortization.
IQS: Average Crop Quality Index (IQS) of the area's farmers (Requires KPI definition).
Competitor Balance: Total volume in Kg registered under the Competitor Estimate field.
Price/Kg: Average price paid per Kg of tobacco received in the Field Technician's area.
Tobacco Flow: Intake/logistics flow indicator.
Current Indebtedness: Real-time financial calculation balancing the farmer's remaining crop season debt against the estimated physical volume of tobacco that is still pending delivery.

Positive & Negative Testing
In this section you can find test scripts for “positive tests” on the end-to-end user processes/capabilities which are goals of this User Story, and the test scripts for “negative tests” which aim is to check if Business Rules are working as expected.
Test Scenarios and Acceptance Criteria
Web
Scenario 1: Dynamic Card Rendering Based on Crop Calendar (Positive)
Test Steps:
Configure the backend so that the Crop Calendar for the 2026 season has the Contracting Phase Active and the Commercialization Phase Inactive.
Log into the web  application using a registered Field Technician account.
Validate the rendering behavior of the Home screen.
Success Criteria: * The Contracting Card must be displayed with all data correctly populated.
The Commercialization Card must not be rendered on the screen.
Scenario 2: Validation of the Commercialization Card Hold Status (Negative)
Test Steps:
Inspect the mobile frontend build package or analyze the Home screen component tree.
Verify if placeholders or API call triggers for the Commercialization Card are deactivated/mocked according to the Hold constraint.
Success Criteria: The system must not trigger network requests to non-existent Buying endpoints, preventing app crashes or unhandled error screens in production while the feature is frozen.
